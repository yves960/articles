# SDD：我发现AI写代码最大的问题不是能力，是约束

## 一个我见过的场景

用户给AI下了一个需求："帮我写一个用户登录功能"。

AI秒出几百行代码。看起来不错？

但仔细一看：
- 用户名长度限制是20个字符（产品要求50个）
- 密码校验规则和安全团队要求不符
- 没考虑手机号登录场景
- 错误提示是英文
- 用了全新的状态管理方案，和项目架构格格不入

用户花了3天修修补补，发现还不如自己从头写。

**这不是AI能力不够，是约束不够。**

AI就像一个超级聪明但没有上下文的新员工。你给模糊指令，它用自己的方式理解，给出一个"正确但不对"的答案。

## 我理解的SDD（规范驱动开发）

SDD的核心只有一句话：

> **先想清楚再动手，用规范约束行为，用验证确保正确。**

传统AI辅助编码：

```
用户描述需求 → AI写代码 → 用户检查 → 发现不对 → 重来
```

SDD的思路：

```
用户描述需求 → 生成规范文档 → AI按规范实现 → 按规范验证
```

**不是让AI自由发挥，而是给AI一张清晰的"施工图"。**

## OpenSpec：让每次变更都有设计文档可追溯

OpenSpec是一个结构化开发工作流框架。核心是一条**Artifact依赖链**：

```
proposal.md → design.md → specs/*.md → tasks.md → 代码实现 → 归档
   (为什么做)   (怎么做)    (做成什么样)  (分几步做)  (动手做)  (存档)
```

每个artifact的职责：

| Artifact | 职责 | 核心内容 |
|----------|------|----------|
| proposal.md | 变更提案 | 为什么做、目标/非目标、影响范围 |
| design.md | 设计决策 | 技术方案选择、替代方案对比、风险评估 |
| specs/*.md | 需求规格 | Requirements + Scenarios（WHEN/THEN） |
| tasks.md | 任务清单 | 可执行的checkbox任务列表 |

**这让我想到了建筑行业。**

你不是让工人直接砌砖，而是先画设计图、结构图、施工图。每一步都有据可查。

## 得物实战：规范体系的三层结构

这是我在得物文章里学到的最有价值的内容。

他们设计了**三层规范体系**，每层解决不同问题：

```
第一层：约束层 → 告诉AI「禁止什么、必须怎样」
第二层：示范层 → 告诉AI「标准产出长什么样」
第三层：视觉层 → 告诉AI「页面应该长什么样」
```

### 为什么需要三层？

**只有约束层时**，AI知道规则但缺乏参考实现，容易在复杂场景下产生符合规则但不符合团队风格的代码。

**加入示范层和视觉层后**，AI可以直接对齐团队的标准产出，减少"虽然合法但不地道"的代码。

### 第一层：约束层

7个规范文件，约束不同维度：

| 文件 | 约束内容 |
|------|----------|
| ts.md | TypeScript规范（禁止any、使用可选链） |
| code-names.md | 命名规范（kebab-case/camelCase/PascalCase） |
| comment.md | 注释规范（JSDoc、@ai-context文件头） |
| lint.md | 代码风格（单引号、文件末尾换行） |
| style.md | 样式规范（Tailwind CSS、less文件） |
| pages.md | 页面目录结构规范（constants/services/hooks/components分层） |
| service.md | API接口生成规范（fetch{Name}Api命名） |

### 第二层：示范层

预置完整的"标准模板代码"：

```
.claude/code-design/
├── pro-table/          # 通用列表页模板
├── pro-form/           # 通用表单页模板
├── editable-pro-table/ # 可编辑表格模板
├── drawer/             # 抽屉组件模板
├── component/          # 通用组件模板
└── utils/              # 工具函数模板
```

**示范代码的作用不只是看格式。**

当开发者说"参考pro-table生成知识治理列表页"时，AI直接继承这套模式，一次就能生成符合团队风格的代码。

### 第三层：视觉层

存放HTML设计稿，覆盖主要页面的视觉参考：

```
.claude/ui-design/
├── knowledge-spaces.html   # 知识空间列表页设计稿
├── search-strategy.html    # 检索配置页设计稿
└── space-detail.html       # 空间详情页设计稿
```

**实践中，提供HTML设计稿后，AI生成的UI与设计意图吻合度明显更高。**

### 实际效果

| 指标 | 结果 |
|------|------|
| 接口命名一致性 | 205个文件保持高度一致 |
| 目录分层 | 每个新页面都正确创建constants/services/hooks/components |
| 代码模板继承 | CRUD页面均参照pro-table模板，结构高度一致 |
| 可选链使用 | 几乎所有数据访问都使用了`?.`和`??`，避免运行时报错 |

## 我发现AI的能力边界

得物的实战让我对AI能力有了更精确的理解。

**不是简单的"能做/不能做"，而是按需求颗粒度选择协作模式：**

### 小颗粒需求：对话框即扫即改

- **场景**：改文案、修显隐逻辑、调CSS间距
- **策略**：直接在Cursor Chat中对话
- **理由**：沟通成本低于编写规范的成本

### 中颗粒标准化需求：基于Rules生成

- **场景**：增加标准CRUD页面、创建简单业务组件
- **策略**：利用预设的Cursor Rules或Skills
- **理由**：这类需求有强烈的"模式感"，规则定义清晰即可高质量输出

### 中大颗粒复杂功能：OpenSpec深度协作

- **场景**：重构核心逻辑、新增复杂业务模块、无参考代码的新功能
- **策略**：OpenSpec标准流（SDD）
- **理由**：业务逻辑复杂时，AI极易产生幻觉或需求偏移，Spec强制"先设计后编码"

## AI失效的三种模式

这是得物实战中最有价值的洞察。

### 模式一：规范真空

任务涉及的领域没有规范约束，AI自行填充"合理默认值"。

- **表现**：代码功能正确，但风格/结构偏离团队约定
- **频率**：高（尤其在新功能开发初期）
- **应对**：在CLAUDE.md或code-design中补充规范，一次修复，全局生效

### 模式二：信息孤岛

AI掌握的信息是当前会话的快照，看不到系统外的状态。

- **表现**：本地正常，CI失败；AI分析每次都对，但解的都是当前暴露的问题
- **频率**：低，但代价高
- **应对**：跨平台、跨环境的依赖要在架构设计阶段提前锁定

### 模式三：任务目标模糊

AI把"该问人的问题"当成"执行问题"来解决。

- **表现**：用户说"优化一下首页"，AI悄悄改了组件结构，而不是先澄清目标
- **频率**：中
- **应对**：Spec工作流的proposal阶段强制要求先描述"Why"

## 典型案例：SDD驱动系统重构

得物分享了一个重构案例，让我印象深刻。

**重构与新功能的根本差异：**

新功能开发是"从无到有"——AI可以大胆生成，错了删掉重来。

重构是"在活体系统上动手术"——不仅要知道改什么，更要知道不能改什么，以及按什么顺序改。

**知识问答首页重构：**

- **架构债务**：大量业务组件与公共组件混放、useChat导出20+方法（4种无关职责混合）、ChatInterface接收17个props
- **执行TASKS**：9组34个子任务，每一步有明确输入和验收标准
- **执行结果**：34个任务全部完成，AI全程独立执行，人工干预<5条指令

**SDD的价值：在动代码之前，把这三件事全部写清楚。**

## 开发者角色的重构

得物文章最后提出一个观点，我很认同：

> **规范设计能力成为AI时代开发者的核心竞争力——能写出让AI可靠执行的规范，价值比能写出同等功能代码更高。**

这意味着：

- **系统性思维更重要**：AI可以解决每个局部问题，但无法帮你看到全局
- **质量意识前移**：过去Code Review在代码写完后进行，现在需要在方案设计阶段就介入
- **规范是杠杆**：AI是力，Spec工作流是支点

## 总结

| 框架 | 一句话 |
|------|--------|
| OpenSpec | "让每次变更都有设计文档可追溯" |
| Superpowers | "让AI代理像有纪律的开发团队一样工作" |
| 得物实战 | "规范三层体系：约束+示范+视觉" |

**AI Coding的本质：用结构化的规范和工作流把不确定性消除在执行之前。**

AI负责在确定性空间里高速执行，人负责维护和扩展那个确定性空间的边界。

**规范是杠杆，AI是力，Spec工作流是支点。**

## 二、SPEC 规范（需求层）

### 2.1 SPEC 文档结构

每个业务需求必须产出 SPEC，存放在项目 `specs/` 目录：

```markdown
# SPEC: [需求名称]

## 元信息
- 编号: SPEC-XXXX
- 作者: 金克丝 (pm)
- 审批: 蔚 (director)
- 状态: draft → review → approved → dev → test → done
- 创建日期: YYYY-MM-DD

## 1. 背景与动机
为什么做这个？用户痛点/业务目标。

## 2. 用户故事
作为[角色], 我想要[行为], 以便[价值]

必填:
- 角色定义
- 核心行为（一个故事一个行为）
- 价值陈述（可衡量）
- 前置条件 / 后置条件

## 3. 功能范围
### In Scope
### Out of Scope

## 4. 验收标准（AC）
每个 AC 为可测试断言:
- Given [前置状态]
- When [触发动作]
- Then [预期结果]

## 5. 技术约束
性能 / 兼容性 / 安全要求

## 6. AI 代码标注
- [AI-GEN] 完全 AI 生成
- [AI-ASSIST] AI 辅助，人工审核
- [HUMAN] 纯人工

## 7. 技术债务记录
| 项目 | 原因 | 优先级 | 计划时间 |
```

### 2.2 用户故事规范

**标准格式：**

```
作为 <角色>,
我想要 <行为>,
以便 <价值>.
```

**好故事 vs 坏故事：**

| ✅ 好 | ❌ 坏 |
|-------|-------|
| 作为注册用户，我想要通过手机号重置密码，以便在忘记密码时恢复账户 | 做一个密码重置功能 |
| 作为管理员，我想要批量导出用户数据为 CSV，以便进行离线分析 | 系统要支持数据导出 |

**拆分原则：**
- 一个用户故事 ≤ 1 个开发工作日
- 超过则拆分为多个故事
- 每个 story 可独立测试和部署

### 2.3 验收标准（AC）规范

**必须满足 INVEST 原则：**
- **I**ndependent — 独立可测
- **N**egotiable — 可讨论实现方式
- **V**aluable — 对用户有明确价值
- **E**stimable — 可估算工作量
- **S**mall — 足够小，一次迭代完成
- **T**estable — 可明确验证通过/不通过

**AC 模板：**
```
AC-1: [标题]
  Given: 用户已登录，账户余额为 100 元
  When: 用户下单购买 50 元商品
  Then: 
    - 订单创建成功
    - 余额变为 50 元
    - 收到订单确认通知
```

### 2.4 技术债务管理

**AI 辅助开发的特殊性：**

AI 生成的代码可能产生特定类型的技术债务：

| 债务类型 | 描述 | 处理策略 |
|----------|------|----------|
| 幻觉代码 | AI 生成了不存在的 API 调用 | 代码审查必查项 |
| 过度工程 | AI 倾向生成复杂的"完美"方案 | 按 YAGNI 原则简化 |
| 模式不一致 | 不同轮次 AI 生成风格不同 | 通过 SPEC 约束代码风格 |
| 缺失边界检查 | AI 常遗漏错误处理 | AC 中必须包含异常场景 |
| 隐式依赖 | AI 可能引入未声明的依赖 | 依赖审查必查项 |

**标注规则：**

每个 SPEC 的产出代码中，技术债务用注释标记：

```python
# TODO(SPEC-0001): [AI-ASSIST] 这里的错误处理需要加强，AI只生成了happy path
# DEBT(SPEC-0001): [AI-GEN] 使用了简化算法，大数据量时需要优化
```

## 三、SKILL 规范（能力层）

### 3.1 Skill 目录结构

```
skills/
└── [skill-name]/
    ├── SKILL.md              # 元数据（必需）
    ├── scripts/              # 可执行脚本
    │   └── main.sh
    ├── references/           # 参考文档
    │   └── api-docs.md
    └── templates/            # 模板文件
        └── config.yaml
```

### 3.2 SKILL.md 元数据规范

```markdown
---
name: [skill-name]
version: 1.0.0
description: 一句话描述何时使用此 Skill
author: 艾克 (dev)
tags: [tag1, tag2]
risk_level: low | medium | high | critical
---

# [Skill 名称]

## 何时使用
触发条件和场景描述。

## 前置条件
- 需要哪些工具/环境
- 需要哪些权限

## 使用方法
具体步骤和示例。

## 输出
执行后产生什么。

## 注意事项
边界条件、已知限制。

## 示例
具体调用示例。
```

### 3.3 Skill 注册与发现

**注册表**（`skills/registry.yaml`）：

```yaml
skills:
  code-review:
    path: skills/code-review/
    owner: dev
    risk: low
    triggers: ["review", "code review", "审查"]
  
  deploy:
    path: skills/deploy/
    owner: devops
    risk: critical
    triggers: ["deploy", "发布", "上线"]
    requires_approval: true
  
  spec-generate:
    path: skills/spec-generate/
    owner: pm
    risk: low
    triggers: ["写需求", "SPEC", "用户故事"]
```

### 3.4 Skill 权限矩阵

| 操作 | pm | dev | devops | director |
|------|:---:|:---:|:------:|:--------:|
| 创建 Skill | - | ✅ | ✅ | ✅ |
| 修改 Skill | - | ✅ | ✅ | ✅ |
| 执行 low risk | ✅ | ✅ | ✅ | ✅ |
| 执行 high risk | ❌ | ⚠️ 需审批 | ✅ | ✅ |
| 执行 critical | ❌ | ❌ | ⚠️ 需审批 | ✅ |

## 四、研发流程（协作层）

### 4.1 完整流程

```
蔚 分配任务
    │
    ▼
金克丝 分析需求 ──→ 输出 SPEC
    │                    │
    │                    ▼
    │              蔚 审批 SPEC
    │                    │
    │              ┌─────┴─────┐
    │              ▼           ▼
    │         艾克 技术方案   凯特琳 测试计划
    │              │           │
    │              ▼           │
    │         艾克 编码实现 ──→│
    │              │           │
    │              ▼           ▼
    │         凯特琳 测试验证
    │              │
    │              ▼
    └──── 蔚 验收发布
```

### 4.2 各阶段详细说明

#### 阶段一：需求分析（金克丝 → 蔚）

**输入：** 蔚转述的业务需求或用户反馈
**输出：** SPEC 文档（按 2.1 模板）

**金克丝的职责：**
1. 理解需求背后的用户情绪和真实痛点
2. 编写用户故事和验收标准
3. 划定功能范围（In Scope / Out of Scope）
4. 提交给蔚审批

**交互方式：** 在群聊中 @蔚 提交 SPEC，蔚审批后标记状态为 approved

#### 阶段二：技术方案（艾克）

**输入：** 已审批的 SPEC
**输出：** 技术方案文档 + 实现代码

**艾克的职责：**
1. 评估技术可行性，必要时 @金克丝 讨论 SPEC 调整
2. 选择/创建合适的 Skill 来实现
3. 编码实现，标注 AI 生成部分
4. 记录技术债务
5. 提交代码并 @凯特琳 进行测试

**交互方式：** 在群聊中或通过 coding-agent 子任务完成编码

#### 阶段三：测试验证（凯特琳）

**输入：** 艾克提交的代码 + SPEC 中的 AC
**输出：** 测试报告

**凯特琳的职责：**
1. 根据 AC 编写测试用例
2. 执行测试，验证每个 AC
3. 检查 AI 生成代码的常见问题（幻觉、边界检查等）
4. 发现问题 @艾克 修复
5. 测试通过后 @蔚 确认发布

#### 阶段四：验收发布（蔚）

**输入：** 凯特琳的测试报告
**输出：** 发布决策

**蔚的职责：**
1. 审核测试结果
2. 确认验收或要求返工
3. 批准发布（凯特琳执行）

### 4.3 Agent 间协作协议

**消息格式：**

```
@接收者 [阶段标签] SPEC-XXXX: 消息内容
```

示例：
```
@艾克 [dev] SPEC-0001: 技术方案需要关注性能，数据量可能到百万级
@凯特琳 [test] SPEC-0001: 代码已提交，请测试 AC-1 到 AC-5
@蔚 [approve] SPEC-0001: 测试全部通过，请审批发布
```

**状态流转：**

```
draft → (金克丝完成编写) → review → (蔚审批) → approved 
→ (艾克开始开发) → dev → (凯特琳测试中) → test 
→ (蔚验收) → done / rejected
```

### 4.4 子任务（Subagent）使用规范

各 Agent 可通过 `sessions_spawn` 创建子任务：

| 发起者 | 子任务类型 | 示例 |
|--------|-----------|------|
| 金克丝 | 调研任务 | 竞品分析、用户反馈收集 |
| 艾克 | 编码任务 | coding-agent 执行实现 |
| 凯特琳 | 测试任务 | 自动化测试执行 |

**子任务规则：**
- 子任务必须关联 SPEC 编号
- 子任务结果自动回报给发起者
- 子任务不跨 Agent（金克丝不创建编码子任务给艾克）

## 六、检查清单

### SPEC 审批检查（蔚使用）

- [ ] 用户故事符合标准格式
- [ ] AC 全部可测试（Given-When-Then）
- [ ] 范围清晰（In/Out Scope 明确）
- [ ] 无模糊表述（"等"、"之类的"）

### 代码提交检查（凯特琳使用）

- [ ] 关联 SPEC 编号
- [ ] AI 生成代码已标注
- [ ] 技术债务已记录
- [ ] 所有 AC 测试通过
- [ ] 无幻觉代码（不存在的 API/库）

### 发布检查（蔚使用）

- [ ] 测试报告无阻塞问题
- [ ] 技术债务优先级可接受
- [ ] 回滚方案已准备

# 如何逐步实现并优化一个 AI Agent Skill

以 flow-trace（跨微服务调用链分析工具）的演进为例

---

## 背景

flow-trace 是一个帮助分析 Java 微服务调用链的 AI Agent Skill。它从简单的代码分析工具起步，经过 30+ 次迭代，逐步演变成一个功能完善、体验流畅的分析助手。这篇文章记录了我们是如何一步步发现问题、解决问题的。

---

## 第一阶段：从 0 到 1

### 初始版本

第一个版本很简单：输入一个入口点，分析代码，输出调用链。功能单一，但能用。

```markdown
# flow-trace
输入入口点 → 分析代码 → 输出 JSON
```

### 第一次重构：AI 驱动

很快发现一个问题：硬编码的分析规则太死板，遇到不同的代码风格就歇菜。于是重构为「AI 驱动模式」——让 AI 自己看代码、理解逻辑，而不是靠正则匹配。

**教训**：AI 擅长理解上下文，不要用传统规则限制了它。

---

## 第二阶段：功能扩展

### 支持网关分析

用户说：「我们的入口是 API 网关，不是业务服务」。于是增加了网关路由解析能力（Spring Cloud Gateway、Kong、Nginx、APISIX）。

### 支持时序图生成

用户说：「文字描述太抽象，能不能画图？」于是增加了 Mermaid 时序图生成。

### 支持表驱动异步流程

这是最难的一个需求：用户说「我们的服务是通过数据库表异步编排的，不是直接调用」。

场景是这样的：
```
服务A: INSERT INTO task_table (status='PENDING')
   ↓ (异步)
服务B: SELECT * FROM task_table WHERE status='PENDING' (定时任务)
```

没有 API 调用，通过表解耦。需要让 AI 识别这种模式，自动搜索读取端。

**实现方式**：
1. 发现 INSERT/UPDATE 时，记录表名和状态字段
2. 搜索 `findByStatus` 方法
3. 搜索 `@Scheduled` 定时任务
4. 找到后继续追踪读取端的下游调用

## 第三阶段：交互优化

### 强制询问机制

最头疼的问题来了：**AI 不听话**。

分析完一个服务后，AI 直接继续分析下一个，根本不问用户。用户说「我想看看中间结果」，但 AI 已经跑远了。

#### 第一次尝试：写在 SKILL.md 里

```markdown
## 注意事项
分析完成后必须询问用户！
```

没用。AI 照样跳过。

#### 第二次尝试：加 🛑 emoji

```markdown
## 🛑 强制规则
🛑 分析完成后必须立即停止！
```

好了一点点，但还是会跳过。

#### 第三次尝试：放进 front matter

```yaml
---
instructions: |
  🛑 分析完成后必须调用 save 命令！
  save 命令会自动输出询问菜单！
---
```

好很多！`instructions` 字段会作为 system prompt，AI 更难忽略。

#### 第四次尝试：脚本强制输出

最终方案：让 `save` 命令执行后**自动输出询问菜单**。

```python
def save_record(...):
    # 保存记录
    ...
    # 强制输出询问菜单
    print("🛑 分析完成！现在必须展示询问菜单！")
    print("1. 分析其他入口点")
    print("2. 深入分析某个节点")
    ...
```

这下 AI 想跳也跳不了了——脚本会强制输出菜单。

**教训**：不要指望 AI 完全遵守 prompt，关键流程要有脚本兜底。

---

## 第四阶段：长对话遗忘问题

用户反馈：分析三四个服务后，AI 就忘了之前分析过什么。

### 解决方案：流程记录脚本

创建了一个 Python 脚本，把每个服务的分析结果持久化到文件：

```bash
# 保存结果
python flow_trace_record.py save user-service UserController.login @/tmp/result.txt

# 查看已分析的服务
python flow_trace_record.py context

# 汇总所有服务
python flow_trace_record.py summary
```

**关键设计**：
- 每次分析前调用 `context`，把之前的分析结果加载到当前上下文
- 分析后调用 `save`，持久化当前结果
- 最后调用 `summary`，汇总所有结果

---

## 第五阶段：代码审视发现问题

有一次让三个 agent（金克丝、艾克、凯特琳）一起审视代码，发现了一个严重 bug：

**`save` 写的是 `result_raw`，但 `list`/`preview` 读的是 `result`**

导致：
- 下游服务提取失效
- Mermaid 图表生成失效
- 未分析服务提醒失效

**教训**：多人审视能发现单人容易忽略的问题。

---

## 第六阶段：持续优化

### 异步调用识别不全

用户测试后发现某些异步逻辑缺失。检查后发现识别规则不够：
- 没覆盖 `@Async` 注解
- 没覆盖 `CompletableFuture`
- 没覆盖 `ApplicationEventPublisher`

于是在 `code-patterns.md` 补充这些模式。

### 菜单选项演进

最初的菜单：
```
1. 分析其他入口点
2. 结束探索
```

用户说「我想深入分析某个节点」，加：
```
3. 深入分析某个节点
```

用户说「异步流程漏了」，加：
```
4. 补充遗漏的异步链路
```

**教训**：菜单选项是根据真实需求逐步演进的，不是一开始就设计完美的。

## 关心的问题

### 1. SKILL.md 多大合适？

一开始只有 50 行，后来涨到 669 行，又精简到 141 行。

**原则**：
- front matter 放强制规则（会作为 system prompt）
- 正文放流程说明和命令用法
- 详细内容放 references/ 目录

### 2. 如何防止 AI 遗忘规则？

三个层次：
1. **front matter instructions**：最难忽略
2. **脚本强制输出**：无法跳过
3. **文档说明**：容易被忽略

### 3. 如何处理跨服务追踪？

- 配置服务目录（config.yaml）
- 脚本 `context` 命令显示已配置路径
- 发现未配置的服务时询问用户

---

## 总结

一个 skill 的演进，不是一开始就设计完美的，而是：

1. **先跑起来**：最小可用版本
2. **听用户反馈**：真实需求在哪里
3. **逐步迭代**：加功能、修 bug
4. **遇到硬伤就重构**：不要怕推翻重来
5. **关键流程要兜底**：脚本比 prompt 更可靠

最重要的是：**不要一个人闷头写，让不同角色审视，能发现很多盲点**。
